# ADR-002: Single Unified Login Flow for All User Roles

## Status

Accepted

## Date

2026-07-03

## Context

Three personas need to log in: end customers, bakery owners, and delivery couriers. The product needs to decide whether they share one login experience or have distinct ones, and how role-based access is determined after authentication.

## Decision

Provide a single, unified login flow (one form, one endpoint) for all personas. Credentials (email + password) are the same shape regardless of role. After successful authentication, the system determines the user's role and returns it alongside the token, so the client can grant access accordingly. No persona-specific login screens or routes.

## Alternatives Considered

### Alternative 1: Role selection before login

- **Description**: User picks "customer / bakery owner / courier" before entering credentials, each option potentially leading to a tailored flow.
- **Pros**: Allows persona-specific messaging or visuals.
- **Cons**: Adds friction with no functional benefit; user could select the wrong option; identity backend is identical either way.
- **Why rejected**: No requirement justifies the added complexity; violates YAGNI.

### Alternative 2: Separate login endpoints per persona

- **Description**: Distinct routes such as `/login/customer`, `/login/bakery`, `/login/courier`, each with its own logic.
- **Pros**: Explicit separation, easy to evolve independently per persona later.
- **Cons**: Duplicates identical authentication logic three times; more surface area to test and maintain for no current business need.
- **Why rejected**: Unnecessary duplication; nothing in the current requirements differentiates the personas at the login step itself, only in what they can access afterward.

## Consequences

### Positive

- One flow to build, test, and maintain.
- Simpler client integration — a single login screen works for all user types.

### Negative

- Role must be resolved and returned as part of (or immediately after) login, since the single flow doesn't get this for free from a persona-specific route.

### Risks

- None significant; role-based authorization enforcement is a separate concern from login itself and can evolve independently.

## Implementation Notes

Left to the TechSpec phase.
