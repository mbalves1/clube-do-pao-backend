# ADR-003: Role Stored in Supabase app_metadata; Per-Table supabaseUserId Link

## Status

Accepted

## Date

2026-07-03

## Context

Codebase exploration for the TechSpec revealed that the three personas (customer, bakery owner, courier) are **not** rows in one shared table. They are three independent Prisma models â€” `User`, `Bakery`, `DeliveryPerson` â€” each with its own creation flow, none with any password or auth linkage today. `DeliveryPerson`'s domain entity (`DeliveryUser`) is an empty stub.

ADR-002 established a single unified login flow. This ADR resolves how that one login endpoint determines which of the three tables an authenticated identity maps to, and how the identity is linked to its business record.

## Decision

1. When a Supabase Auth credential is created for a person (customer, bakery owner, or courier), their role (`customer` | `bakery_owner` | `courier`) is written to that Supabase user's `app_metadata` at creation time.
2. Each of `User`, `Bakery`, and `DeliveryPerson` gets a new unique `supabaseUserId` column linking the business row to its Supabase Auth identity.
3. On login, the backend calls Supabase `signInWithPassword`, reads `role` directly from the returned user's `app_metadata` (no extra query needed to know the role), then looks up the matching business record in the table implied by that role, using `supabaseUserId`.

## Alternatives Considered

### Alternative 1: Query all three tables after authentication

- **Description**: After Supabase authenticates the credential, query `User`, then `Bakery`, then `DeliveryPerson` by a linking column until a match is found.
- **Pros**: No dependency on Supabase metadata being set correctly at account-creation time.
- **Cons**: Up to three sequential database round trips per login; no single source of truth for role; ambiguous if a linking column is ever set on more than one table for the same identity.
- **Why rejected**: Slower and more fragile than reading a value already returned by the authentication call itself.

### Alternative 2: Merge User, Bakery, and DeliveryPerson into one table with a role discriminator

- **Description**: Collapse the three models into a single `Person`/`Account` table with a `role` enum column.
- **Pros**: Single table, simplest long-term data model for identity.
- **Cons**: Large, high-risk migration touching every existing relation (`FavoriteBakery`, `Subscription`, `Order`) across three unrelated business concepts (customers, bakeries-as-businesses, delivery couriers); far beyond what the PRD scoped; violates YAGNI for a login feature.
- **Why rejected**: Business records for a bakery (a company) and a delivery person are conceptually different from a customer account; merging them is an unrelated, much larger data-modeling project.

## Consequences

### Positive

- Login resolves role and business record with exactly one Supabase Auth call plus one targeted database lookup.
- No ambiguity: role is set explicitly and once, at credential-creation time.
- Existing `User`, `Bakery`, `DeliveryPerson` tables and their relations are untouched aside from one new nullable/unique column each.

### Negative

- Role changes (if ever needed) require updating Supabase `app_metadata`, a second system, rather than a single database update.
- `Bakery` and `DeliveryPerson` rows created before this feature have no `supabaseUserId` and cannot log in until one is linked (accepted â€” see PRD Non-Goals and the companion onboarding decision).

### Risks

- The current Supabase client in `src/middlewares/auth.ts` is initialized with the service role key, used so far only for `getUser(token)` verification. Calling `signInWithPassword` from a service-role-initialized client is expected to work (GoTrue's password grant only requires a valid `apikey` header, not specifically the anon key), but this must be verified against the project's actual Supabase instance during implementation. Mitigation: if it fails, add `SUPABASE_ANON_KEY` as a new documented environment variable and use it only for the sign-in call.

## Implementation Notes

- `supabaseUserId` is unique and required (`String @unique`) on `User` since every customer registers with a credential going forward. On `Bakery` and `DeliveryPerson` it is unique and optional (`String? @unique`) since those records are not all linked to a credential yet.
