# ADR-004: Backend-Proxied Token Refresh

## Status

Accepted

## Date

2026-07-03

## Context

Access tokens issued by Supabase Auth are short-lived (the project default, per ADR discussion during PRD: standard Supabase expiration with background renewal). Clients need a way to obtain a new access token using a refresh token without forcing the user to log in again. The project currently only has `SUPABASE_SERVICE_ROLE_KEY` documented as an environment variable (`docs/infra.md`); there is no `SUPABASE_ANON_KEY` wired into any client-facing configuration.

## Decision

The Express backend exposes `POST /api/auth/refresh`, accepting a `refreshToken` and forwarding it to Supabase Auth's session-refresh call using the existing service-role-initialized client. It returns a new access token, refresh token, and expiration. Clients never talk to Supabase directly.

## Alternatives Considered

### Alternative 1: Client refreshes directly against Supabase using the anon key

- **Description**: Frontend/mobile clients hold the Supabase anon key and use the Supabase SDK to refresh sessions without involving the Express backend.
- **Pros**: Removes one hop; offloads refresh traffic from the backend; is the pattern Supabase's own SDKs are designed around.
- **Cons**: Requires introducing and documenting a new environment variable (`SUPABASE_ANON_KEY`) and distributing it to every client; is a bigger change to the project's current all-requests-go-through-Express convention.
- **Why rejected**: No client currently holds any Supabase key; centralizing through the backend matches how the rest of the API already works and avoids a new distribution/configuration concern for an MVP.

## Consequences

### Positive

- Clients only ever talk to the existing Express API, consistent with every other feature.
- No new environment variables or client-side Supabase configuration needed for the MVP.

### Negative

- Backend proxies refresh traffic, adding a small amount of load and one extra network hop compared to a direct client-to-Supabase call.

### Risks

- None significant; this can be revisited later if client-side direct refresh becomes desirable.

## Implementation Notes

Left to implementation; reuses the Supabase client already instantiated in `src/middlewares/auth.ts` (or a shared instance extracted for both auth verification and sign-in/refresh calls).
