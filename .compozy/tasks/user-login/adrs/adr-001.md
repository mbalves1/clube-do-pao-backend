# ADR-001: Use Supabase Auth as the Identity Provider

## Status

Accepted

## Date

2026-07-03

## Context

The project needs a login/signin capability so registered users can authenticate and receive a token for subsequent requests. Three user personas need to log in: end customers, bakery owners, and delivery couriers. A future phase must also support "Sign in with Google."

Today, user registration only stores `name` and `email` in the local `User` table — there is no password or credential of any kind, so no one can currently log in. The project's documented stack (`docs/stack.md`) already designates Supabase for authentication, and `authMiddleware` already verifies Supabase-issued JWTs for other protected routes.

The alternative under consideration was building a separate, project-owned credentials table (storing hashed passwords) and issuing custom JWTs, entirely independent of Supabase.

## Decision

Use Supabase Auth as the single identity provider for all user personas. Registration will be updated to create the credential via Supabase Auth (email + password) in addition to the existing local `User` profile row. Login will authenticate against Supabase Auth and return the Supabase-issued access token. Google sign-in (Phase 2) will use Supabase Auth's native OAuth provider support, reusing the same identity system.

## Alternatives Considered

### Alternative 1: Custom credentials table + custom JWT issuance

- **Description**: Store password hashes in a project-owned table (e.g., `bcrypt` hash on `User` or a new `Credential` table) and hand-roll JWT issuance, verification, and refresh.
- **Pros**: Full control over token contents and expiration policy; no dependency on Supabase Auth's user model.
- **Cons**: Duplicates functionality Supabase already provides (hashing, token issuance, refresh tokens, expiration); requires building and maintaining a second identity system when Google sign-in is added later, since Google OAuth would need to be reconciled with the custom credentials table; contradicts the documented stack decision; `authMiddleware` already expects Supabase-issued tokens and would need to be reworked or duplicated.
- **Why rejected**: Higher implementation and maintenance cost for no added business value, and directly conflicts with the existing stack decision and Google sign-in roadmap.

## Consequences

### Positive

- Login, token issuance, refresh, and future Google OAuth all share one identity system.
- Reuses the Supabase client and verification logic already present in `authMiddleware`.
- Less code to build and maintain long-term.

### Negative

- Registration flow must change to create a Supabase Auth user (password required), which is a breaking change to the current signup contract (name + email only).
- Any user created before this change has no Supabase Auth credential and cannot log in until a credential exists for them (out of scope for this PRD — see Open Questions).

### Risks

- Existing users without a Supabase Auth credential will be unable to log in. Mitigation: tracked as an open question for a follow-up migration/backfill decision, not blocking this PRD.
